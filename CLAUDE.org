** Animation Architecture (IMPORTANT - Keep Consistent!)
*** Structure
All animation code lives in =boneglaive/graphical/animations/=. One file per unit.

File organization:
- =core.py= - Base classes (Particle, AnimatedUnit, DebrisParticle, ParticleEmitter)
- =animation_factory.py= - Maps skill names to animation classes
- =glaiveman.py= - All GLAIVEMAN skill animations
- =potpourrist.py= - All POTPOURRIST skill animations
- =mandible_foreman.py= - All MANDIBLE FOREMAN skill animations
- =__init__.py= - Package exports (update when adding new animations)

*** Two Animation Patterns
1. *Object-based*: Self-contained animation objects (projectiles, effects)
   - Example: =SpinningGlaiveProjectile=, =LightningBolt=
   - Has =update(delta_time)= → returns =True= if still active
   - Has =draw(surface)= → renders to pygame surface
   - Created by factory, managed by renderer's =active_animations= list

2. *Attribute-based*: Manipulates unit properties directly
   - Example: =PryAnimationController= (launches unit, manipulates position)
   - Controller object manages unit attributes (=unit.pry_phase=, =unit.pry_timer=)
   - Has =update(delta_time)= → returns =True= if still active
   - Has =draw(surface)= → usually empty (unit renders itself)
   - Created by factory, managed by renderer's =active_animations= list

*** CRITICAL: Animation update() Return Value
*ALL ANIMATIONS MUST RETURN True/False FROM update()!*

The renderer checks the return value to determine if animation should continue:
#+begin_src python
still_active = anim.update(delta_time)
if still_active:
    updated_animations.append(anim)  # Keep in active list
#+end_src

*Common Bug*: Returning =None= (implicit return) causes animation to disappear immediately!

*Correct Pattern* (see EstrangeBeam for reference):
#+begin_src python
def update(self, delta_time):
    self.timer += delta_time

    if self.phase == "charge":
        # ... phase logic ...
        if self.timer >= self.charge_duration:
            self.phase = "next_phase"
            self.timer = 0

    elif self.phase == "final_phase":
        if self.timer >= self.final_duration:
            return False  # Animation complete

    return True  # Animation still active
#+end_src

*Issue found in*: Neural Shunt animation (2025-11-22) - was returning None, causing immediate removal from active_animations.

*** Renderer Responsibilities (NO ANIMATION LOGIC!)
=renderer.py= should ONLY:
- Create animations via =AnimationFactory.create_animation()=
- Update animations: =for anim in self.active_animations: anim.update(delta_time)=
- Draw animations: =for anim in self.active_animations: anim.draw(surface)=
- Remove finished animations from list

*** Damage and Game Logic
- Animations are VISUAL ONLY - no damage application
- Game logic in =skills.py= handles all damage/HP changes
- Damage numbers queued during animations, shown when animations complete
- See =pending_animation_events= and =flush_pending_events()= in renderer

*** Pre-Execution Blocking Animations (NEW!)
*Problem*: Some skills move units DURING execution (knockback, displacement), but we want animations to show effects at ORIGINAL positions before movement.

*Solution*: Pre-execution blocking system allows animations to play BEFORE game logic executes.

*Implementation* (in =renderer.py=):
1. =PRE_EXECUTION_BLOCKING_SKILLS= list (line ~57) - Skills that block execution
2. =execute_turn()= checks for blocking skills after pre-sync (line ~1873)
3. If detected: flushes animations immediately, waits for completion, THEN executes turn
4. =_update_animations_only()= helper updates visuals without game state sync

*Adding Blocking Skills*:
#+begin_src python
# In renderer.py (top level, near color constants)
PRE_EXECUTION_BLOCKING_SKILLS = [
    "Fragcrest",      # Shows cone burst before knockback moves units
    "FRAGCREST",      # Normalized variant
    "YourSkill",      # Add your skill here
    "YOUR_SKILL"      # Include normalized version
]
#+end_src

*When to Use*:
- Skills that displace/move targets (knockback, pulls, pushes)
- Skills where visual order matters (effect THEN movement)
- Skills that need targets at original positions for visual clarity

*Example*: Fragcrest cone burst fires, shrapnel hits targets, THEN units are knocked back. Without blocking, units would already be at new positions when animation plays.

*Performance*: Blocking loop runs at 60fps, updating only animations/visuals. No game state syncs during wait.

*** Adding New Animations
1. Create animation class in appropriate unit file (e.g., =glaiveman.py=)
2. Add to =animation_factory.py= SKILL_ANIMATIONS dict
3. Handle special constructor args in factory's =create_animation()= if needed
4. Export class in =animations/__init__.py=
5. Test with =python run_graphical.py=

*** Anti-patterns (DON'T DO THIS!)
❌ Animation logic scattered in =renderer.py= update loop
❌ Separate file per animation (use one file per unit instead)
❌ Animations applying damage/healing
❌ Special-case animation handling outside factory pattern

* Key Design Decisions
- *Turn-based engine*: Players alternate, actions resolved in sequence
- *Greek letter unit IDs*: Alpha, Beta, Gamma, etc. for unit identification
- *Status effects*: Complex system with timers, immunities, interactions
- *Event system*: Decoupled communication via =utils/event_system.py=
- *Double-buffered rendering*: Reduces terminal flicker in curses
- *JSON serialization*: For network messages and save data

* Dependencies
- Python 3.x with curses, json, socket, threading (standard library only)
- No external package dependencies

* Common Patterns and Troubleshooting

** Adding New Status Effects in Graphical Mode

When a new status effect is added to a skill, THREE places must be updated for graphical mode to work properly:

1. *boneglaive/graphical/ui/status_effects.py* - Status effects panel
   - Add entry to STATUS_EFFECTS dict (~line 27-180)
   - Define: name, type (buff/debuff/special/neutral), icon text, description, duration_key, check lambda
   - This makes the status appear in the UI panel when a unit is selected

2. *boneglaive/graphical/game_state.py* - Status effect detection
   - Add attribute check in VisualUnit._get_status_effects() (~line 82-140)
   - Pattern: =if hasattr(game_unit, 'attribute_name') and game_unit.attribute_name:=
   - Map to icon filename (without .svg): =effects['icon_name'] = True=
   - This triggers the SVG icon flash animation when effect is applied

3. *graphics/status_icons/* - SVG icon file
   - Icon file must exist with matching name (e.g., =site_inspection.svg=)
   - Animation system loads this automatically via StatusIconFlash class

*** Example: Site Inspection Status Effect (2025-11-23)

*Problem*: Site Inspection skill applied stat changes, panel showed the status, but SVG icon didn't flash on unit tile.

*Root Cause*: Missing attribute checks in =game_state.py:_get_status_effects()=

*Fix Applied*:
#+begin_src python
# boneglaive/graphical/game_state.py line ~131-134
if hasattr(game_unit, 'status_site_inspection') and game_unit.status_site_inspection:
    effects['site_inspection'] = True
if hasattr(game_unit, 'status_site_inspection_partial') and game_unit.status_site_inspection_partial:
    effects['site_inspection_partial'] = True
#+end_src

*Lesson*: When status effect panel works but icon doesn't flash, check =_get_status_effects()= in =game_state.py=. The icon name in this dict MUST match the SVG filename (without .svg extension).
